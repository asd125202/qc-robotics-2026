Pitch 40 · Compliance CI/CD For Physical AI

CertForge

不是写合规，是铸证据。CertForge 把每一次 Dragonwing 机器人 release 的代码、模型、SBOM、ROS/MCAP、HIL/真机测试、QNN profile、风险评估和供应商证明，自动编成可审计技术文件证据链。

01 · Problem

机器人公司死在量产前夜，不是因为不知道要合规，而是因为没有证据系统。

软件可以持续发布，机器人不行。每一次模型、固件、传感器、无线模组、电池、外壳、速度限制、说明书或供应商变化，都会让安全、网络、无线、电气、AI 和市场准入证据重新失效。

版本风险

每次更新都可能破坏合规假设

QNN 模型重编译、ROS 节点升级、SBOM 漏洞、传感器换型和 OTA 策略改变，都需要重新判断影响范围。

证据散落

证据散在十几个系统

Git、Jira、Confluence、实验室 PDF、仿真报告、ROS bag、供应商邮件和 SBOM 工具之间没有同一个证据链。

末端审查

实验室通常看到的是最后快照

认证顾问和测试机构能审查样机，却很难追回每次设计决策、风险接受、变更影响和真实运行证据。

采购审查

企业客户买的是可审计能力

采购、EHS、保险、法务和 IT 安全团队需要看到证据包，而不是一句“我们以后会做认证”。

平台缺口

芯片能力没有变成准入能力

Qualcomm 的安全启动、AI profile、功耗、连接和生命周期优势，需要被包装成量产和采购能理解的证据。

02 · Current Alternatives Fail

现有工具各管一段，没有人负责“机器人 release 能不能卖”。

实验室、PLM、ALM、SBOM、仿真、GRC 和认证顾问都重要，但它们没有围绕 Dragonwing 机器人版本构建一条从设计到现场的证据图谱。

实验室 / 顾问

擅长末端测试和解释路径，但通常太晚介入，无法自动追踪每个工程变更的影响。

PLM / ALM

管理需求、物料和流程，却不天然理解 ROS/MCAP、模型 profile、现场事件和机器学习版本。

SBOM / SAST

能回答软件供应链问题，无法回答机械危险源、功能安全、无线、电池和运行场景风险。

Simulation

能产生大量测试数据，但如果没有法规条款、危险源和签核关系，数据不会自动变成审计证据。

通用合规自动化

Saphira AI、Normal、Ketryx 等证明市场正在形成；CertForge 的切口是机器人 evidence graph + Dragonwing release integration。

03 · Solution

给机器人团队一条合规 CI/CD：从版本发布自动生成可审计证据。

CertForge 不替代认证机构、测试实验室或律师。它连接研发、测试、运行和供应链系统，维护一张图：法规条款 -> 危险源 -> 控制措施 -> 测试 -> 证据 -> 签核人 -> release。

Classify

选择机器人类型、用途、目标市场、无线、电池、电源、环境、用户和应用边界。

Map

生成适用法规、标准、测试路径、不适用理由和需要人类签核的判断点。

Collect

从 Git/CI、BOM、SBOM、QNN profile、ROS/MCAP、仿真、HIL 和真机测试收集证据。

Gate

每次模型、固件、供应商、无线、电池、外壳、风险策略和说明书变化触发 impact review。

Export

输出 EU、US、中国、客户采购、保险尽调和实验室沟通所需的 evidence dossier。

04 · Why Now

2026-2028 是机器人从 demo 走向监管证据的窗口期。

Physical AI 正在同时遇到新硬件平台、新 AI 部署方式和更严格的网络安全、机械安全、无线设备和 AI 文档要求。越早把证据自动化，越容易从开发板走到量产。

2025 EU RED 网络安全要求已从 2025-08-01 开始适用于相关无线设备。
2026 EU CRA 漏洞和严重事件报告义务从 2026-09-11 开始。
2027 EU Machinery Regulation 从 2027-01-20 开始适用，CRA 主要义务从 2027-12-11 开始。
AI AI Act 让高风险 AI 的文档、日志和治理压力逐步进入实体产品。
ISO ISO 10218-1/-2:2025 更新，AMR、协作和移动机器人安全证据继续细化。

05 · Product

产品不是“合规问答机器人”，而是机器人证据操作系统。

CertForge 把 Dragonwing 机器人每次 release 的上下文保存成可复用、可查询、可签名、可导出的证据节点。评委能看到一个真实产品团队会每天使用的系统，而不是一次性的申报文档。

  • Evidence Graph：版本化追踪风险、控制措施、测试、签核和 release。
  • Robot Run Recorder：把 ROS2/MCAP、仿真、HIL、真机测试转换成场景覆盖和安全证据。
  • Cyber + AI Pack：SBOM、VEX、漏洞报告、模型卡、数据 lineage、QNN profile 和回归门禁。
  • Technical File Export：Machinery、CRA、AI Act、RED、ISO 10218/3691-4、客户审计和 gap ticket。

06 · Evidence Artifacts

每个输出都对应一个真实买家、审计人或实验室问题。

我们不说“自动认证”。我们说：让认证、采购、保险、融资和客户安全审查需要的材料，在研发过程中持续生成。

requirements-matrix.json

目标市场、适用标准、条款映射、不适用理由和签核记录。

evidence-node.json

证据类型、产品版本、关联条款、哈希、附件、签名和状态。

qnn-model-evidence.json

模型哈希、QNN backend、量化差异、profile、边界和安全相关性。

sbom-vex-pack.zip

组件清单、漏洞状态、影响判断、修复计划和客户安全问卷证据。

change-impact-report.json

某次变更影响哪些需求、风险、测试、文档、说明书和上市包。

launch-dossier-index.pdf

给实验室、客户、投资人和 Qualcomm 生态伙伴看的证据目录。
