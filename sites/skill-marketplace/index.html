Pitch 03 · Trusted Robot Skill Distribution

SkillDock

机器人技能不能像普通 App 一样随手安装。SkillDock 不是模型仓库，而是把 LeRobot lineage、SkillCertKit、EdgeRuntimeBench、Qualcomm edge profile、SBOM、安全边界、分阶段发布和回滚做成同一张可信技能卡。

从信任问题开始 连接 SkillCertKit

01 · Problem

机器人行业的复用断在最后一公里。

同一个抓取、巡检、分拣、质检技能，每个客户、每台机器人、每套相机和每个 runtime 都要重新集成。开发者无法把技能卖第二次，买家也不敢把陌生模型装到真实机器人上。

痛点 1

技能没有可迁移身份

模型、ROS 节点、配置、相机要求和安全边界散落在项目文件里，离开原团队就很难复用。

痛点 2

买家看不到安装风险

企业买家关心的是兼容 robot profile、延迟、功耗、权限、网络依赖、失败模式和回滚计划。

痛点 3

开发者没有第二次销售

好技能经常被埋在一次性集成项目里，无法变成可定价、可支持、可升级的商品。

痛点 4

平台缺少默认目标

没有统一 edge profile，技能市场会变成截图目录；有 Qualcomm profile，才有可验证的运行边界。

02 · Why Now

现在的缺口不是有没有模型，而是技能能不能可信分发。

UR+、MiR Go、ROS Index、Viam Registry、Intrinsic Flowstate、Reachy apps 和 OpenMind OM1 已经证明机器人生态会走向 marketplace、registry 和 skill catalog。但它们大多绑定 OEM、runtime 或开发工具。新机会在于：跨机器人 profile 的 AI skill evidence。

需求已验证

机器人买家接受认证生态

UR+ 和 MiR Go 说明企业客户愿意为降低部署时间、成本和风险购买经过测试的组件和应用。

供给已出现

LeRobot 让技能来源可追踪

TeleopStudio、CloudTwin 和 TrainRouter 可以把 episode、训练、评估和模型版本连成可审计 lineage。

目标已成熟

Qualcomm 给技能可验证边缘目标

AI Hub、QNN/QAIRT、Device Cloud、Dragonwing profiles 和 Robotics Hub 可以让技能卡带上真实运行证据。

03 · Solution

SkillDock 的核心不是 SKU 数量，而是信任密度。

一个技能能卖，是因为买家能提前看到：在哪些 robot profile 上测试过，来自哪些数据，在 Qualcomm edge 上延迟、内存、功耗如何，安全边界是什么，失败后怎么回滚。

Capture

TeleopStudio 生产 quality-gated episodes，保留 operator、camera、state/action 和失败接管。

Train

CloudTwin 和 TrainRouter 生成 policy、eval report、dataset lineage 和 runtime candidate。

Profile

EdgeRuntimeBench 在 Qualcomm target 上记录 latency、memory、runtime、数值正确性和资源边界。

Certify

SkillCertKit 生成 manifest、SBOM、license、signature、safety boundary、policy decision 和 rollback。

Publish

SkillDock 上架、销售、私有审批、分阶段安装、升级、遥测和回滚。

04 · Product

技能卡不是宣传页，而是安装前 admission contract。

每个技能以 digest-addressed evidence bundle 分发。`skill.yaml` 是入口，后面挂接 hardware profile、runtime constraints、dataset/model lineage、edge profile、SBOM、签名、policy decision 和 rollback plan。

  • 身份：skill id、版本、digest、证据等级和支持方。
  • 边界：任务、传感器、动作频率、工作空间、禁用动作和接管规则。
  • 兼容：robot profile、Qualcomm target、OS、runtime、相机、夹爪和控制器。
  • 交付：安装门禁、分阶段发布、上一稳定版本、健康检查和回滚触发器。

05 · Market & Business Model

机器人买一次，技能、验证、发布和证据每年续费。

切入点不是所有机器人，而是已有安装基础里的高频复用技能：教育实验室、系统集成商、机器人 OEM、SME 自动化试点、RaaS fleet operator、仓储/工厂/实验室运营方。

Marketplace Ready

上架验证费

按 skill、robot profile、硬件族或 Qualcomm target 收取 readiness / validation fee。

Private Library

企业私有技能库

私有技能、审批流、SSO/RBAC、审计日志、版本冻结、采购资料和离线安装。

Runtime

每机器人运行订阅

签名安装、版本控制、遥测 hook、分阶段 rollout、rollback 和支持 SLA。

Channel

SI / OEM 收入分成

行业技能包、系统集成服务、OEM profile 和 RaaS fleet co-selling，而不是替代渠道。

06 · Competition & Moat

不能说“第一个机器人 App Store”，要说“证据先于安装”。

已有生态证明市场方向：UR+、MiR Go、ROS Index、Viam Registry、Reachy apps、OpenMind OM1、NVIDIA Isaac/GR00T、Intrinsic Flowstate、Unitree SDK、KUKA/ABB/RoboDK 工具生态。SkillDock 的差异是跨 profile 的安装前证据和 Qualcomm-first edge readiness。

短期壁垒

精选目录与 manifest schema

先做少数高频技能：sample transfer、视觉抓取、AMR 巡检、目标跟随、轻量质检。

中期壁垒

compatibility graph

每次安装、阻止、回滚、现场失败和人工接管都沉淀为 skill x robot x sensor x runtime 的兼容数据。

长期壁垒

skill evidence network

拥有“技能 x 数据来源 x robot profile x Qualcomm target x 现场验收”的网络，越多技能通过越难复制。

07 · Why Qualcomm

SkillDock 可以把 Dragonwing 从硬件选项，变成技能分发默认目标。

SkillDock 的 marketplace rule 很简单：没有命名的 Qualcomm edge profile、runtime evidence、兼容边界和 rollback path，就不能作为可信技能发布。这样 AI Hub、QNN/QAIRT、Device Cloud、Qualcomm Linux 和 Robotics Hub 不再是分散工具，而是技能生态的证据后台。

Profile 技能卡绑定 RB3 / QCS6490 / QCS8550 / IQ profile 和传感器要求。
Compile AI Hub / QNN / QAIRT 输出 compile、profile、context 和 runtime 证据。
Gate 安装前检查 latency、memory、permissions、SBOM、safety 和 rollback。
Flywheel 更多技能瞄准 Dragonwing，更多机器人选择 Dragonwing 以获得技能目录。
需要高通支持

Dragonwing Skill Profile

共同定义轻量 schema：target chipset、sensor、control rate、runtime、model hash、latency、memory、evidence level 和 rollback。

需要高通支持

AI Hub / QNN office hour

跑通一个策略组件从 PyTorch/ONNX 到 AI Hub profile、QNN/QAIRT artifact 和 ONNX Runtime QNN fallback。

需要高通支持

2-3 个 reference skills

在 Robotics Hub 与 SkillDock 展示 vision grasping、AMR inspection 或 visual tracking 的 profiled skill card。

08 · Demo & Evidence

3 分钟演示要证明：陌生技能不能直接装，可信技能可以分阶段发布。

比赛演示不需要完整交易市场，而要展示一个真实技能卡如何阻止错误安装、允许兼容安装、运行后回滚，并把失败数据回流到 CloudTwin。

0:00-0:45

打开技能卡

展示 `labforge.sample-transfer.v1` 的 task boundary、LeRobot lineage、SkillCertKit 和 EdgeRuntimeBench 证据。

0:45-1:30

阻止错误安装

选择不兼容 robot profile，policy gate 给出具体阻止原因：相机、runtime、workspace 或安全边界不匹配。

1:30-2:15

分阶段发布

选择兼容 Qualcomm profile，允许 staged rollout 到一台设备，展示 latency、memory、signature 和 rollback plan。

2:15-3:00

失败回滚

模拟 safety event 或 health check 失败，回滚到 previous-known-good，并把失败 episode 送回 CloudTwin。

SkillDock 的一句话：机器人技能市场不是下载按钮，而是“证据先于安装”的商业分发层。
