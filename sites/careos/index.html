Pitch 50 · Eldercare Robotics Productivity Platform

给护理团队，多一双会学习的手。

护元 CareOS 是面向居家、社区和机构养老的机器人照护操作系统：让护理员把巡护、提醒、递送、风险上报和人工接管交给安全边界内的 Qualcomm 端侧机器人，把每次失败变成下一版 LeRobot 技能。

看我们怎么解决 看商业模式

01 · Problem

养老行业不是缺一个聊天机器人，而是缺可复制的护理产能。

护理员的时间被夜间巡房、重复提醒、找物送物、家属沟通和服务记录切碎；老人和家属真正担心的是安全、及时响应、隐私和可追溯。机构、社区和长护险服务商需要的是可证明的照护过程，不是一台会表演的机器人。

人手不够

全球护理人力供给跟不上老龄化；中国失能失智、高龄独居和机构照护需求持续增加。

时间被切碎

巡护、提醒、递送、取物和记录是高频任务，却很难形成稳定服务质量。

家属不放心

家属需要摘要和异常通知，但老人生活空间不能变成默认上传的监控视频。

服务难结算

长期护理服务需要过程证据、任务记录、异常处置和可审计的服务闭环。

02 · Why Now

三个变化同时发生：需求爆发、支付成形、端侧机器人终于可用。

如果只是宏观老龄化，这不是创业机会；真正的窗口来自政策支付、服务网络和边缘 AI 能力在同一时期成熟。

3.2338 亿 2025 年末中国 60 岁及以上人口，占总人口 23.0%。
40.6 万 2024 年中国养老机构和设施数量，已形成服务部署入口。
长期护理 长护险制度化推进，未来智能化服务和支持性辅具有望进入服务包。

03 · Insight

养老机器人第一性原理：先放大护理员，而不是替代护理员。

海外市场已经验证机构物流、陪伴提醒、跌倒/活动监测、药盒提醒都有付费场景，但它们是孤岛。真正的大产品不是“机器人护士”，而是把这些窄任务接入同一个人机协同、可接管、可训练、可审计的护理操作系统。

不是陪聊

ElliQ 证明陪伴有需求，但 CareOS 的核心是任务闭环和服务证据。

不是送货

Moxi、Relay、Aethon 证明机构物流可行，但 CareOS 解决养老连续照护。

不是全能人形

Labrador 的启发是务实：先递送、提醒、受限操作，再学习更多照护动作。

04 · Solution

护元 CareOS 把护理碎片任务变成一条可学习的照护流水线。

护理员创建或确认任务，机器人在安全边界内执行提醒、递送、巡护和轻物品操作；低置信度或异常立即升级给人；服务记录进入审计账本，接管片段进入 LeRobot HIL 数据集。

  • 对护理员：少跑腿、少漏提醒、少重复记录。
  • 对机构/社区：服务过程可复盘、可验收、可结算。
  • 对家属：看到授权摘要和异常通知，而不是全天候监控。
  • 对 OEM/Qualcomm：形成可复制的养老机器人参考设计和技能包。

05 · Product Workflow

一个任务从护理站出发，最后回到训练闭环。

页面和比赛 demo 要让评委看到实际使用流程，而不只是架构名词。

Request

护理员或系统创建 CareTask：送水、提醒、巡护、轻物品递送。

Verify

机器人本地识别房间、人物姿态、托盘物品、二维码和隐私区域。

Assist

低速导航，执行边界清晰的提醒、递送、取物或护理站跑腿。

Escalate

无回应、低置信度、动作失败和安全事件升级给护理员或远程接管。

Learn

失败与接管变成 LeRobot episode，训练后灰度发布到 Qualcomm edge。

06 · Market

先打养老机构和社区长护服务，不从纯 C 端硬件零售开始。

Airbnb 的早期 pitch 清楚地先找到 beachhead。CareOS 也一样：第一批客户不是所有老人，而是有预算、有流程、有服务记录需求的机构、社区站、长护险服务商和 OEM。

中国机构

养老机构、护理院、康复机构、连锁养老集团，以夜间巡护和护理站跑腿切入。

中国社区

街道养老站、居家床位、长护险定点服务商，以服务记录和异常上报切入。

海外机构

senior living、assisted living、home-care agencies，以 workforce support 和 RaaS 切入。

OEM 平台

机器人厂商购买 CareOS SDK、Qualcomm edge profile 和 Skill Cloud 任务包。

07 · Business Model

收入一句话：按“可验收的照护能力”收费，而不是一次性卖硬件。

硬件是入口，复购来自软件、服务记录、运维 SLA、技能包和场景数据闭环。

试点包

3-6 个月、3-10 台机器人，含调研、部署、培训、运维和验收报告。

机构 RaaS

硬件租赁 + CareOS 软件 + 运维 + 数据看板，按台/月和任务包分层。

居家长护包

服务商主导，机器人 + 家属小程序 + 远程看护 + 服务记录。

OEM SDK

CareOS runtime、Qualcomm profile、Skill Cloud、任务包和认证材料授权。

08 · Competition

现有产品证明需求，但没有把任务、接管、证据和训练连起来。

CareOS 不否认竞品价值。相反，它把已验证的窄场景统一成可部署的平台层。

ElliQ / Hyodol

陪伴和提醒强，物理任务与机构工作流弱。

Labrador

日常递送方向务实，但缺少机构级任务和训练平台。

Moxi / Relay

机构物流成熟，但主要面向医院和固定路线。

Nobi / Vayyar

跌倒和活动监测明确，但不是可行动机器人。

CareOS

用 Qualcomm edge + CareTask + HIL 把感知、动作、接管和数据闭环连接起来。

09 · Architecture

技术架构放在商业逻辑之后：本地可运行、隐私可证明、失败可接管。

CareOS 用 Qualcomm edge 执行实时感知和安全动作，用 Care Cloud 管任务、证据和角色，用 Skill Cloud 训练和发布可回滚技能。云端训练不能绕过本体侧安全监督。

Robot Edge

RB3 Gen 2 / IQ-9075 / IQ10、端侧推理、断网缓存、签名 OTA、secure boot。

Privacy

默认上传事件、姿态和位置摘要，不默认上传原始生活空间视频。

CareTask

FHIR Task-style 状态机，记录任务、确认、异常、家属摘要和服务证据。

Action Shield

轻物品、低速、受限工作空间、力/速度限制、急停和人工接管。

Nav Safety

Nav2 keepout、collision monitor、卧室/卫生间隐私区、楼梯/厨房风险区。

LeRobot HIL

低置信度、动作失败和远程接管进入 episode，训练后灰度发布。

10 · Competition Demo

比赛演示一条完整照护闭环，不碰医疗红线。

用模拟养老房间、二维码药盒、毛巾/水杯托盘、隐私区地图和一次失败接管，展示从任务创建到服务证据再到训练数据的完整路径。

Map

标记卧室、卫生间、厨房、楼梯、护理站和 keepout zones。

Task

创建送水/提醒任务，机器人扫描二维码做核验。

Assist

机器人低速递送水杯或毛巾，只做辅助，不做给药。

Escalate

模拟无回应或取物失败，机器人停机并请求护理员接管。

Train

接管片段生成 LeRobot HIL episode，输出 Qualcomm edge artifact。

11 · Why Qualcomm

养老是 Qualcomm 机器人参考设计最适合落地的长期场景之一。

养老机器人需要低功耗、多摄像头、多传感、端侧推理、弱网运行、隐私边界和长生命周期支持。CareOS 把这些硬能力包装成居家、社区和机构都能理解的照护生产力。

Edge AI

本地姿态、物体、条码、障碍、任务核验和隐私过滤。

Security

secure boot、signed OTA、mTLS、ROS 2 enclaves、SBOM/VEX。

Training

LeRobot HIL 数据集、AI Hub profile、灰度部署、回滚。

Ecosystem

参考设计、OEM SDK、试点 benchmark、护理机器人开发者案例。

Ask

用 Qualcomm 开发板完成安全边界 demo，并联合定义养老机器人参考任务包。
