Pitch 26 · Qualcomm-first Robot Runtime OS

RobotCoreOS

机器人公司不缺一次能跑的 Demo，缺的是可交付、可更新、可回滚、可审计的运行底座。RobotCoreOS 把 Qualcomm 开发板变成一套面向商业化机器人的 golden runtime image。

进入路演 连接 SkillCertKit

01 · Problem

机器人商业化最脏、最贵、最容易被低估的部分，是系统维护。

每个团队都在重复自建 OS image、BSP、驱动、ROS 2 launch、相机/NPU 配置、日志、远程诊断、OTA、回滚、证书、客户隔离和模型发布门禁。Demo 可以成功一次，但客户现场需要 20 台、50 台、200 台机器可以被长期运营。

版本不可追溯

现场到底跑哪个 kernel、driver、ROS graph、模型、技能包和参数，常常没有一份可信记录。

更新像赌博

模型、驱动、镜像和配置都可能破坏任务。没有灰度、健康门禁和 A/B 回滚，OTA 会变成现场风险。

边缘 AI 难交付

云上训练出的 policy 要落到本体，还要处理 QNN/QAIRT、输入规格、fallback、温度和功耗。

买方怕运维

企业客户买的不是酷 demo，而是 uptime、审计、恢复、责任边界和安全发布流程。

集成商被拖慢

系统集成商每个项目都从镜像、驱动和远程支持重新开始，交付成本吞掉毛利。

生态难复用

没有标准 runtime contract，技能、日志、模型、硬件 profile 和 fleet ops 很难跨项目复用。

02 · Why Now

具身智能正在从“能跑一次”进入“能部署一批”。

ROS Noetic 已在 2025 年 5 月到达 EOL，ROS 2 Jazzy LTS 面向 Ubuntu 24.04 时代；LeRobot 让真实数据采集和策略训练更开放；Qualcomm Dragonwing、AI Hub、QNN/QAIRT 和 Device Cloud 正在补齐边缘 AI 工具链。现在缺的是把云训练结果安全地放到机器人本体上的产品层。

ROS 维护窗口

商业队列需要冻结过的 LTS image、迁移计划、补丁策略和可验证的系统生命周期。

RaaS 经济模型

Robot-as-a-Service 把 CapEx 变成 OpEx，也把 uptime、远程诊断、proof-of-work 和服务 SLA 变成采购核心。

边缘模型发布

策略模型不再只是文件，而是带输入、输出、硬件、性能、权限、安全边界和回滚包的发布单元。

Qualcomm 机会

低功耗、多摄像头、连接和端侧 AI 很适合真实机器人队列，但生态需要一个默认 runtime target。

03 · Solution

把开发板变成可出货机器人，需要一套黄金运行镜像。

RobotCoreOS = Qualcomm board profile + validated drivers + ROS 2 bridge + LeRobot hooks + QNN/QAIRT policy runner + SafetyOps + signed OTA + A/B rollback + fleet identity + runtime manifest。

  • 它不替代 ROS 2，而是把 ROS 2 放进可维护的产品运行环境。
  • 它不替代 Viam、Formant、Foxglove，而是给这些云层提供可信本体 endpoint。
  • 它不只是 OTA agent，而是绑定系统镜像、技能包、模型、数据和安全边界的 runtime contract。
  • 它把 Qualcomm edge profile 写进每次部署证据，让硬件价值进入产品机制。

04 · Product Workflow

从工厂烧录到现场回滚，每一步都留下证据。

RobotCoreOS 的产品价值不是一个镜像文件，而是一条从 hardware bring-up 到 fleet ops 的交付流程。客户拿到的是可启动、可验证、可更新、可恢复的机器人底座。

Flash

烧录 golden image，绑定 SoC、BSP、kernel、driver、传感器和 recovery profile。

Enroll

首次开机完成 device identity、客户归属、证书、网络和 fleet enrollment。

Validate

检查相机、IO、IMU、CAN/UART、急停、NPU/GPU/CPU placement 和 ROS lifecycle。

Install

安装通过 SkillCertKit 的 policy package，记录模型 hash、输入输出、权限和回滚包。

Operate

本地执行策略、安全边界、日志、episode capture、telemetry 和异常上报。

Profile

输出 latency、FPS、memory、accelerator utilization、温度、功耗和 runtime manifest。

Update

对 image、app、skill、model 和 config 做签名更新、灰度发布和健康门禁。

Rollback

失败启动、性能退化或安全门禁失败时自动回到上一稳定版本或安全模式。

Capture

把失败片段、低置信度、人工接管和任务 episode 回流到 LeRobot 数据闭环。

Audit

导出 SBOM、签名、发布记录、硬件 profile、模型 lineage 和现场恢复报告。

05 · Market

第一批客户不是“所有机器人公司”，而是最怕现场失控的人。

这个产品的 wedge 是从 Qualcomm 开发板到量产机器人之间的 productization gap。买方愿意付费，是因为它减少系统团队、现场返工、版本事故、集成工时和客户采购疑虑。

机器人 OEM

VP Engineering 需要可复现的 BSP、Linux、ROS、driver、OTA baseline，避免长期养发行版团队。

系统集成商

需要 installable image、已知可用的 ROS/driver stack、远程支持和回滚，减少每个项目的脏活。

RaaS 运营商

按月对 uptime 负责，需要 identity、telemetry、proof-of-work、staged rollout 和故障恢复。

企业 IT / 安全

关心设备身份、补丁 SLA、SBOM、CVE、审计日志、访问控制和 fleet inventory。

教育与开发套件

需要学生和开发者能刷机、恢复、运行 ROS/LeRobot 示例，而不是先修嵌入式 Linux。

行业试点客户

工厂、实验室、物流、零售和园区试点需要 5-50 台机器先被可靠运营起来。

06 · Business Model

卖的不是“一个 Linux 镜像”，而是机器人从 Demo 到队列的维护能力。

定价需要高于 generic IoT device management，低于完整 RobOps 平台；核心是按设备授权加企业订阅，再用 OEM/NRE 和集成商包做早期现金流。

Community

免费开发镜像、示例 profile、课程和论坛支持，种子用户来自高校、比赛和开发套件。

Runtime

每台机器人授权，包含 signed image、ROS 2 baseline、rollback OTA、device identity、SBOM 和 update channel。

Fleet

按机器人/月订阅，包含分组、灰度、远程诊断、日志、审计、版本治理和 incident bundle。

OEM / SI

面向厂商和集成商收取 bring-up、secure boot、私有镜像、长期维护、认证 profile 和项目模板费用。

07 · Competition

真正的竞争不是某一个产品，而是四类工具拼起来的临时方案。

RobotCoreOS 的定位必须清楚：ROS-compatible，不是 ROS replacement；Qualcomm-first，不是通用云 dashboard；bootable runtime image，不只是 OTA agent。

ROS 2

默认机器人中间件和工具生态，但不是 secure product OS、fleet updater、rollback layer 或发布证据系统。

Ubuntu Core / Yocto

提供生产 OS 和嵌入式构建基础，但不定义机器人技能、ROS topic contract、LeRobot lineage 和 SafetyOps。

Mender / Balena

验证 A/B OTA 和设备管理是刚需，但机器人还需要 motion safety、policy manifest、episode 数据和硬件 profile。

Viam / Formant

验证 fleet ops、telemetry、remote control 和 RobOps 市场，但不是 bootable Qualcomm BSP/runtime layer。

Foxglove

验证机器人数据和可视化的重要性；RobotCoreOS 应默认输出 clean MCAP、ROS bag、telemetry 和 incident bundle。

NVIDIA Isaac

证明 physical AI 需要完整栈；RobotCoreOS 的差异化是 Qualcomm-native、低功耗、camera-rich、connected edge。

08 · Moat

护城河来自运行证据，而不是某个单点功能。

每一次 bring-up、失败 OTA、传感器漂移、温度降频、人工接管、技能安装和客户恢复，都会沉淀成下一版 runtime rule、compatibility profile 和发布门禁。

Profile Qualcomm board、driver、camera、IO、accelerator 和传感器矩阵。
Evidence SBOM、provenance、签名、OTA、回滚、health gate 和审计记录。
Lineage 模型、技能、LeRobot dataset、episode、失败片段和版本历史。
Network OEM、集成商、教育套件、行业模板和 fleet incident 数据不断复用。

09 · Runtime Evidence

每个 golden image 都必须带一份 release evidence bundle。

评委和客户需要看到这不是“又一个机器人 OS 名字”，而是一套可验证的交付规范。比赛阶段可以先模拟流程，复赛拿到板卡后补实测 profile。

Image

不可变基础系统、独立 data 分区、signed image digest、board profile、recovery path、reproducible build log。

Security

device identity、证书、密钥注入、secure boot/FDE 可用性说明、无 secrets baked into image。

SBOM

系统、容器、应用、模型和技能包生成 SPDX/CycloneDX，附 vulnerability、license、VEX/exception notes。

Lifecycle

ROS 2 managed nodes、watchdog、health gate、safe mode、network allowlist、DDS/SROS 2 权限。

Policy

声明模型 hash、runtime target、输入输出、max rate、speed/force/zone、依赖和 rollback package。

Telemetry

导出 latency、FPS、memory、power、thermal、accelerator placement、safety hits、update state 和 rollback cause。

10 · Why Qualcomm

RobotCoreOS 让 Qualcomm 从“被使用的一块板”变成机器人技能的默认边缘目标。

Qualcomm 的价值不是只报 TOPS，而是低功耗边缘 AI、多摄像头/多媒体、连接、端侧隐私、弱网韧性和长期硬件生态。RobotCoreOS 把这些能力写进 runtime manifest、edge profile、发布门禁和 demo 证据。

Dragonwing-first

从 RB3 Gen 2 / QCS6490 入口，扩展到 QCS8550、IQ 系列和更高阶机器人计算。

AI Hub / QNN

模型包记录 AI Hub compile/profile/inference 结果、QNN/QAIRT 版本、backend 和数值验证。

Profiler

demo 不只展示“能跑”，还展示 latency、memory、utilization、FPS/W、thermal headroom 和 throttling 前后表现。

Ecosystem

可成为 Dragonwing Robotics Hub 的 reference runtime project，帮助开发者从 prototype 走向 production。

11 · Demo Story

三分钟证明：这不是 Demo 机器，而是可交付机器。

比赛初赛先用模拟 dashboard 和本地运行样例展示完整闭环；复赛拿到开发板后补真实 Qualcomm profile、功耗、温度、延迟、视频和失败回滚证据。

Boot

开机进入 RobotCoreOS console，展示 board profile、image digest、ROS 2 baseline 和 device identity。

Install

安装一个 LeRobot policy package，展示 model manifest、输入规格、权限和 safety boundary。

Run

运行桌面机械臂或移动底盘任务，实时显示 latency、camera、ROS topics、accelerator placement。

Fail

触发错误 OTA 或性能退化，health gate 阻止发布，系统回滚到上一稳定版本。

Export

导出 runtime manifest、失败 episode、SBOM、release record、NPU/温度/功耗 profile 和审计报告。

12 · Ask

给 Qualcomm 的请求：把 RobotCoreOS 做成 Dragonwing 机器人 runtime 样板。

我们希望得到 6-8 周 Dragonwing validation sprint：RB3 Gen 2 或指定开发板、AI Hub/Device Cloud quota、QNN/QAIRT 指导、profiling review、机器人生态伙伴引荐和可公开的 reference project 机会。

  • 共同定义 3 个 Qualcomm-first benchmark：vision pipeline、multi-model sensor fusion、small VLM/LLM robot assistant。
  • 统一报告 latency、memory、accelerator utilization、FPS/W、thermal headroom、power 和 rollback success。
  • 把比赛成果整理成中文 pitch deck、developer tutorial、runtime manifest schema 和 demo video。
  • 所有性能数字标注日期、SDK、板卡、散热、电源、模型版本和是否模拟。

13 · Claim Boundaries

边界讲清楚，可信度才会提高。

RobotCoreOS 不是安全认证系统，不是硬实时控制器，不替代合规安全控制，也不声称已经获得 Qualcomm 官方认证。它的可 defend claim 是：一套 Qualcomm-first、可签名、可更新、可回滚、可观测、可审计的机器人运行镜像和发布证据流程。

RobotCoreOS 把“这块板能跑 Demo”升级成“这台机器人可以被交付、更新、回滚和长期维护”。
