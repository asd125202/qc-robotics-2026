RobotMac Core

01 · Problem

机器人团队不缺开发板，缺的是能从 demo 走到现场部署的产品底座。

开发板解决“能跑”，不解决“能卖、能维护、能扩张”。进入 5-100 台 fleet 后，真正拖垮团队的是电源、相机、I/O、ROS、更新、日志、回滚、安全边界和企业 IT 问答。

Power

机器人不是 USB 设备

12/24V、brownout、reverse polarity、motor/logic rail isolation、E-stop、watchdog 和 thermal 必须产品化。

I/O

每个本体接口都不同

cameras、USB、Ethernet、CAN、UART、PWM、GPIO、IMU、LiDAR、ToF 和 time sync 每次重做。

Runtime

ROS 能跑不等于能交付

ROS 2 lifecycle、DDS QoS、SROS2、driver health、logs、device identity 和 local safety state 需要统一边界。

AI

模型不是部署包

LeRobot policy 还要经过 ONNX、AI Hub/QNN/QAIRT、runtime profile、fallback、签名和 rollback。

Fleet

试点后才暴露运维成本

OTA、SBOM、audit trail、remote diagnostics、fleet update 和 field recovery 常常不是第一版考虑。

Buyer

客户买的是可恢复能力

买方担心版本不可追溯、更新不可恢复、安全边界不清、现场没人能修。

02 · Current Alternatives Fail

现有方案都重要，但没有把机器人从 prototype 带到 fieldable product。

RobotMac Core 不替代开发板、ROS、fleet tools、OTA tools、整机平台或 PLC。它把这些能力组织成一个 robot-native deployment appliance。

SBC / Dev Kit

适合原型，但 enclosure、power、camera、safety I/O、OTA、rollback 和 fleet ops 仍要自己做。

ROS 2

事实标准中间件，但 install drift、driver quality、networking、security 和 observability 仍然难。

RobOps Tools

Viam、Formant、Foxglove、InOrbit 强在云和数据，但通常假设可靠 robot computer 已经存在。

IoT OTA

Balena、Mender、Ubuntu Core、Yocto 不理解 robot action space、safety gate 和 autonomy rollback。

Robot Bodies

Unitree、AgileX、Husarion、Clearpath 本体强，但每个 body 都有不同 compute/runtime 差异。

PLC / Controllers

确定性控制和 safety 强，但不是围绕 LeRobot/VLA/AI Hub/cloud training loop 设计。

03 · Solution

RobotMac Core 是 dev kit 和全定制 electronics 之间的 robot compute box。

它的承诺不是 raw TOPS，而是减少 prototype 到 field fleet 的集成时间和现场失败：known-good hardware、RobotCoreOS、AI edge path、safety boundary、fleet evidence 和 pilot handoff。

Hardware

12-24V input、brownout、camera carrier、CAN-FD、USB3、2.5GbE、M.2、debug/service port。

RobotCoreOS

Ubuntu/Yocto base、ROS 2 LTS、lifecycle services、device identity、signed A/B OTA、rollback。

AI Edge

LeRobot -> ONNX -> AI Hub compile/profile/validate -> QNN/QAIRT or ONNX Runtime QNN where verified。

Safety Boundary

Linux/ROS/AI 不作为 certified safety controller；E-stop、watchdog、motor enable 走独立边界。

Evidence

OpenTelemetry、ROS diagnostics、MCAP/logs、runtime manifest、SBOM、episode capture、incident ledger。

05 · Product

产品体验应该像 appliance 开箱，而不是工程板 bring-up。

RobotMac Core 把硬件、OS、ROS、AI runtime、identity、OTA、telemetry 和 safety boundary 组合成可验收 runtime slot。

  • Core S：QCS6490-class，12 TOPS，12-24V，2.5GbE，CAN-FD，USB3，M.2，2-3 validated cameras。
  • Core Vision：Core S + CSI/GMSL camera carrier、sync trigger、IMU、calibrated camera bundle。
  • Core Pro：QCS8550-class，48 INT8 TOPS target，16GB RAM，4-6 cameras，higher thermal envelope。
  • Core Dev：RB3/Q6A/Rhino/VENTUNO-compatible SDK image 和 harness，只卖开发，不假装量产 appliance。
  • Core Max：IQ10-based roadmap/reference design exploration，不作为 2026 年 7 月近期开售 SKU。

06 · Product API

客户买到的不是盒子，而是可验收的 robot runtime slot。

所有指标标注来源：`validated-on-board`、`AI-Hub-device-cloud`、`proxy`、`simulated`、`product-target` 或 `future-profile`。这比假装所有能力都已量产更可信。

board-profile

SoC、board、carrier、OS、kernel、BSP、camera stack、runtime versions、image digest。

power-io-manifest

12/24V rails、motor/logic isolation、E-stop、watchdog、CAN/UART/GPIO/PWM、camera topology。

runtime-slot

ROS graph、LeRobot dataset schema、policy hash、model runtime、permissions、safety envelope。

edge-profile

AI Hub/QNN/QAIRT/ONNX path、latency p50/p95、memory、temperature、power、fallback labels。

fleet-identity.sbom

device identity、cert chain、SBOM、container/image digest、CVE review status。

ota-rollback-record

bundle signature、cohort、install result、mark-good、rollback pointer、previous-known-good。

07 · Market & Business Model

不按 cheap SBC 定价，按避免几个月集成和一次现场失败定价。

第一批买家是 robot OEM、SI、RaaS/fleet operator、教育/比赛团队和海外 robotics startup。中国版强调硬件速度和出口支持，海外版强调 security、lifecycle、documentation、support 和 lower integration risk。

中国版

机器人 OEM + SI + 教育/比赛

AMR、service robot、lab automation、industrial inspection、清洁、具身智能创业公司。卖点是缩短 bring-up、形成可复用线束/runtime、支持中文双云训练和本地 SI 交付。

海外版

机器人创业公司 + fleet operator + integrator

卖点不是低价，而是 secure OTA、SBOM、fleet audit、lifecycle support、edge evidence、support SLA 和 cert-readiness。
