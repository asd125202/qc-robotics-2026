Pitch 05 · Qualcomm-first Robotics Workbench

DragonWorks

机器人团队不缺单点工具，缺一条从 Dragonwing 设备、LeRobot 数据、双云训练、AI Hub/QNN 证据、SkillCertKit 门禁到 SkillDock 发布的可复现流水线。DragonWorks 把这些证据放在一个 Qualcomm-first 工作台里。

从平台断层开始 打开交互原型

01 · Problem

机器人开发卡在“从 demo 到可复现产品”的断层。

设备注册、遥操作数据、训练任务、模型版本、边缘 profile、技能证据、fleet 日志和比赛提交材料分散在不同工具里。结果是 demo 能跑，但团队无法回答：这次成功来自哪个设备、哪份数据、哪个训练 job、哪个 Qualcomm runtime，以及失败后如何回滚。

断点 1

设备和数据脱节

相机、传感器、执行器、runtime、firmware 和 LeRobot episode 没有统一 device lineage。

断点 2

云训练和边缘目标脱节

团队能启动 GPU job，却常常忘记训练目标应该从第一天就绑定 Qualcomm profile。

断点 3

模型和技能交付脱节

一个权重文件离可安装技能还差 edge metrics、SBOM、权限、安全边界、签名和回滚。

断点 4

比赛和商业交付脱节

视频、截图、指标、源码、BOM、证据表和答辩故事没有自动从真实项目状态生成。

02 · Why Now

Qualcomm 已经有工具链碎片，现在需要一条开发者旅程。

Dragonwing Robotics Hub、AI Hub Workbench、QNN/QAIRT、Device Cloud、Profiler、Qualcomm Linux 和 Dragonwing/RB3/IQ reference designs 都在形成。机会不是再造 ROS 或仿真平台，而是把这些资产串成从真实机器人数据到可部署技能的 Qualcomm-first journey。

开发者时机

LeRobot 让真实数据进入标准工作流

TeleopStudio 和 CloudTwin 可以把 episode、失败片段、训练任务和评估报告变成可追踪资产。

边缘时机

AI Hub / QNN 可以成为 release gate

模型不只要训练出来，还要有 compile、profile、runtime、latency、memory 和 fallback 证据。

生态时机

NVIDIA 已经讲清 Isaac 路径

Qualcomm 也需要一条同样清晰的机器人路径：real data -> cloud train -> edge profile -> skill release。

03 · Solution

DragonWorks 是把真实机器人 episode 变成 Qualcomm edge skill 的控制平面。

它不替代 ROS 2、LeRobot、AI Hub、Foxglove、Viam 或 Formant。它站在这些工具之上，把项目、设备、数据、训练、边缘证据、技能门禁、发布和比赛提交组织成一个可复现 release graph。

Device

注册 Dragonwing / QCS / IQ target、robot profile、camera、actuator、runtime 和健康状态。

Dataset

接入 TeleopStudio / LeRobot episode，锁定 dataset hash、split、失败片段和数据地域。

Train

调用 TrainRouter，选择 China / Overseas lane、预算、GPU provider、eval gate 和 target profile。

Profile

进入 AI Hub / QNN / QAIRT / ONNX Runtime QNN 路径，生成 Qualcomm edge evidence。

Release

通过 SkillCertKit，上架 SkillDock，接入 EdgeFleet staged rollout、日志、失败回流和回滚。

04 · Product

工作台最重要的屏幕不是 dashboard，而是 evidence packet。

DragonWorks 把分散在各页面的证据聚合成一次 release：从 device registry 到 dataset manifest，从 cloud job 到 Qualcomm export，从 skill gate 到 install audit。

  • `device-registry.json`：board、SoC、OS、runtime、sensor、actuator、firmware、health。
  • `job_contract.v1.yaml`：dataset、policy、target profile、budget、lane、eval、export。
  • `qualcomm_export_manifest.json`：ONNX/QNN/QAIRT artifacts、AI Hub job、runtime、input shape。
  • `release-record.json`：skill digest、policy decision、staged rollout、previous-known-good 和 rollback pointer。

05 · Market & Business Model

免费入口吸引开发者，工作流证据向团队和企业收费。

第一批用户是 Qualcomm/Arduino/Hackster-style 开发者、机器人课程、实验室和比赛团队；付费客户是需要复现、团队协作、私有数据、边缘 profile、fleet evidence 和技能上架材料的 OEM、SI、RaaS 和企业创新团队。

Developer

免费项目层

设备模板、dataset card、训练 job、demo storyboard 和 submission builder，降低进入门槛。

Team

团队协作层

按 seat、设备数、私有数据集、训练 job、profile credits 和 release history 收费。

Enterprise

企业证据层

私有部署、SSO/RBAC、审计、regional data lanes、SkillCertKit evidence 和 EdgeFleet hooks。

Ecosystem

生态收入层

reference skills、SI/OEM templates、SkillDock 上架、validation services 和 training credits。

06 · Competition & Moat

DragonWorks 不要和 Isaac、Flowstate、Viam、Foxglove、Formant、ROS 正面互换定位。

Isaac 是 GPU-first simulation/physical AI stack；Flowstate 是工业 workcell 平台；Viam/Formant 更偏 robot runtime 和 fleet ops；Foxglove 是 data observability；LeRobot 是 open robot-learning stack；ROS 是底层机器人基础设施。DragonWorks 的位置是 Qualcomm-first release journey。

短期壁垒

Qualcomm target templates

RB3/QCS6490、QCS8550、IQ-8275、IQ10 路线的 device profile、AI Hub profile 和 demo templates。

中期壁垒

release evidence graph

每个 episode、training job、edge profile、skill gate、install、failure、rollback 都沉淀为可查询关系图。

长期壁垒

ecosystem distribution

开发者模板、Robotics Hub reference projects、SkillDock skills、SI/OEM channels 和 competition submissions 互相强化。

开发者不应该只在最后一步才想到部署硬件。DragonWorks 让 Qualcomm target 从项目创建开始出现：数据采集时记录 board/sensor/runtime，训练时绑定 target profile，发布前生成 AI Hub/QNN evidence，安装后回收 EdgeFleet failure data。
